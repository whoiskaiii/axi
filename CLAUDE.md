# AXI 课程项目规范

## 1. 目标

本项目基于 `pulp-platform/axi` 仓库编写一套 30 课的 AXI 深度教程。

`CLAUDE.md` 的作用不是介绍项目背景，而是作为持续写课时必须遵守的执行标准。所有后续课件、答案、补充说明，均应以本文件为准。

本项目的核心目标如下：

- 输出一套以源码为锚点的 AXI 课程，而不是脱离仓库的泛泛协议讲解。
- 保持课程内容长期一致，避免术语混乱、图表风格漂移、章节质量不稳定。
- 让每一课都既能解释协议概念，也能落到本仓库中的具体实现。
- 让文档在纯文本环境下仍然具备良好的可读性。

## 2. 强制规范

### 2.1 语言与写作风格

- 教材主体语言为中文，但必须保留必要英文术语。
- 整体风格采用“教科书风格”：概念讲解、源码引用、图表说明、课后练习四者并重。
- 论述应以清晰、准确、可核对为优先，不追求花哨表达。
- 每个重要结论都应落到协议、源码或两者之一，不能只做抽象表述。
- 说明代码时，必须解释“它解决什么问题”“关键字段/逻辑是什么”“为什么这样设计”。

### 2.2 中英混排规范

- 协议名、信号名、类型名、宏名、模块名、接口名、枚举名必须保留源码原文。
- 对行业内更自然的英文术语，优先保留英文，不强行直译。
- 术语首次出现时，使用“中文解释 + 英文原词”或“英文原词 + 中文解释”。
- 后续上下文清楚时，可直接使用英文术语。
- 图中标签、表格字段名、代码讲解中的标识符，优先使用源码原名。
- 不要为了追求全文纯中文而发明生硬译法。

默认允许直接保留英文的术语包括：

- `burst`
- `handshake`
- `backpressure`
- `throughput`
- `outstanding`
- `arbiter`
- `scoreboard`
- `struct`
- `packed struct`
- `parameter type`
- `typedef`
- `interface`
- `modport`

以下名称默认不翻译：

- `AW/AR/W/B/R`
- `valid/ready`
- `req_t/resp_t`
- 仓库中的字段名、宏名、模块名、类型名

### 2.3 源码引用规范

- 每一课至少引用 1 处本仓库源码。
- 文件路径与行号必须真实可核对，不能凭记忆编造。
- 优先引用本仓库源码，其次引用 `CONTRIBUTING.md` 等仓库规范，再考虑外部协议。
- 一段分析尽量对应一个明确的源码锚点。
- 若一个结论依赖多处实现，应先给主锚点，再给辅助锚点。
- 引用源码时，保持以下格式：

```systemverilog
// src/axi_pkg.sv, 第 75-87 行
localparam BURST_FIXED = 2'b00;
```

### 2.4 文件与命名规范

- 课程正文文件命名为 `tutorial/lessons/Lesson_XX_Title.md`。
- 课程答案文件命名为 `tutorial/lessons/Lesson_XX_Answers.md`。
- `XX` 必须是两位数字编号。
- 文中术语、图中标签、表格字段、代码标识符必须前后一致。
- 同一概念不得混用多套叫法，除非明确说明它们同义。

## 3. 课程模板

每一课默认采用以下结构。若某课确有必要调整，可合并相邻小节，但不得缺失核心信息。

### 3.1 固定章节

1. 课程信息头
2. 学习目标
3. 目录
4. 正文内容
5. 源码映射
6. 关键术语表
7. 课后练习
8. 下一课预告

### 3.2 课程信息头

课程信息头应包含：

- 课程系列
- 所属单元
- 前置知识
- 学习时长
- 对应源码

### 3.3 正文内容要求

- 正文必须围绕“本课解决什么问题”展开。
- 每节正文都应同时覆盖概念解释与源码落点。
- 若本课涉及结构定义、宏展开、接口拼接、方向归属、位宽计算等内容，应配具体示例。
- 若本课涉及事务过程、握手过程、burst 行为、响应返回等动态行为，应给出事务过程说明，必要时配 WaveDrom。
- 若本课内容本质是对比，应优先使用表格，而不是拆成大量零散段落。

### 3.4 练习设计要求

- 每课练习建议为 4-6 题。
- 练习难度应递进。
- 至少覆盖以下类别中的两类：
  - 概念理解
  - 代码阅读
  - 推导分析
  - 位宽计算
  - 宏展开
  - 仓库定位
- 练习必须能由正文推出，不得考正文未铺垫的内容。
- 答案文件不能只给结论，必须给出推导过程。

### 3.5 写课工作流

每次编写新课程时，默认按以下顺序执行：

1. 先确认本课主题在总大纲中的位置，以及与前后课程的衔接关系。
2. 先在仓库中定位本课最关键的源码锚点，再开始写正文，避免先写概念后找代码。
3. 先列出“本课解决什么问题”和学习目标，再展开正文。
4. 正文优先写清概念主线、源码落点、关键示例，再补图表和术语表。
5. 图表选择必须遵循作图规范，不因美观偏好改变工具优先级。
6. 练习题在正文完成后设计，确保题目能由正文推出，并覆盖本课关键能力点。
7. 答案文件在练习题确定后单独编写，给出推导过程，不只给结论。
8. 提交前按“验收清单”逐项自检，再统一清理术语、命名、重复表述和图表风格。

## 4. 作图规范

### 4.1 工具选择优先级

本项目中图表工具的默认优先级为：

`WaveDrom > ASCII art > Markdown 表格 > Mermaid`

`Mermaid` 仅作为补充方案，不作为默认方案。

### 4.2 WaveDrom 使用规则

- 所有波形图、时序图、握手图、读写事务时序图、burst 过程图，统一使用 `WaveDrom`。
- 必须使用 ` ```wavedrom ` 代码块包裹。
- 不要用 ASCII art 或 Mermaid 代替时序图。

### 4.3 ASCII art 使用规则

- 框图、结构图、分层图、模块连接图默认使用 `ASCII art`。
- 凡是表达依赖严格对齐、等宽排版、并列比较、总线相对位置、左右镜像关系的图，必须使用 `ASCII art`。
- 图必须在纯文本环境下仍然可读。

### 4.4 Mermaid 使用规则

- `Mermaid` 仅用于对齐要求不高的简单关系图、简单流程图、FSM 草图。
- 如果 `Mermaid` 自动布局导致节点漂移、列不齐、语义关系变弱，则不要使用。
- 不要把已经排版良好的 ASCII 框图机械替换为 `Mermaid`。

### 4.5 表格使用规则

- 字段对比、位宽统计、通道差异、宏参数说明等内容，优先使用 Markdown 表格。
- 本质上是行列表达的内容，不强行画图。

### 4.6 图表通用要求

- 一张图只表达一个核心关系。
- 图中命名优先与源码一致。
- 必要时在图下补一句中文解释，说明读图方法或关键结论。

## 5. 验收清单

完成一课前，必须逐项自检：

- 本课是否明确回答了“解决什么问题”。
- 是否给出了清晰的学习目标。
- 是否至少引用了 1 处真实源码锚点。
- 每个核心结论是否都有协议或源码落点。
- 是否至少包含 1 个具体示例。
- 是否至少包含 1 个核心图表。
- 图表工具是否符合作图规范。
- 术语首次出现时是否已定义。
- 若保留英文术语，是否给出了必要中文解释。
- 代码讲解是否解释了设计意图，而不是仅贴代码。
- 源码映射章节是否覆盖了本课关键文件/宏/模块/类型。
- 练习是否能由正文推出。
- 练习难度是否递进。
- 若涉及时序、握手、方向或 burst，是否给出了清晰的事务过程说明。
- 文中命名、图中标签、表格字段、代码标识符是否前后一致。
- 是否清理了重复表述、空泛总结、与源码不一致的说法。

## 6. 仓库参考

### 6.1 关键目录

- `tutorial/lessons/`：课程正文与答案
- `tutorial/AXI_30_Lessons_Syllabus.md`：30 课总大纲
- `tutorial/AXI_Learning_Guide.md`：学习路线和整体建议

### 6.2 常用源码锚点

- `src/axi_pkg.sv`：协议常量、类型、辅助函数
- `src/axi_intf.sv`：AXI4/AXI4-Lite `interface`
- `include/axi/typedef.svh`：通道 `struct` typedef 宏
- `include/axi/assign.svh`：`interface` 与 `struct` 间赋值宏
- `src/axi_test.sv`：验证组件
- `CONTRIBUTING.md`：仓库编码与接口设计约定

### 6.3 项目背景约束

- 本仓库遵循 *AMBA AXI and ACE Protocol Specification, Issue F.b*
- 编码风格遵循 lowRISC SystemVerilog Coding Style Guide
- 模块依赖分为 Level 0-6，课程内容应尽量按层级递进
- `struct` 端口风格是仓库核心设计约定，`_intf` 变体主要承担连线适配

## 附录：禁写清单

- 不要脱离本仓库泛泛而谈协议背景。
- 不要只讲抽象概念而不给源码落点。
- 不要只贴源码而不解释设计意图。
- 不要把代码逐行翻译成中文。
- 不要为了凑篇幅反复改写同一个结论。
- 不要发明生硬、不自然、业内少用的中文译法。
- 不要在图中使用与源码不一致的命名。
- 不要把适合表格表达的内容强行画图。
- 不要把依赖严格对齐的结构图强行改成 `Mermaid`。
- 不要写未经验证的行号、位宽、字段数量、接口方向。
- 不要省略前提条件，例如宽度参数、拍数、默认值、方向归属。
- 不要把练习写成与正文脱节的额外知识问答。
- 不要只给练习结论，不给推导过程。
- 不要加入低信息量、仅为形式存在的章节。
